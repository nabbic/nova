How does the seller connect?

This is the hardest UX and trust problem in the product. How do we get seller systems accessible?

A

Lightweight on-prem agent

Seller installs a small agent inside their network. Agent runs locally, collects read-only data, and sends structured results to the platform. Nothing opens an inbound port — agent calls out.

Pros

  • Works for private networks, air-gapped, on-prem infra
  • Seller sees exactly what's collected
  • No credentials leave the network

Cons

  • Requires seller IT involvement to install
  • Adds friction in a time-pressured process
B

Cloud connectors (OAuth / API tokens)

Seller authorises integrations via OAuth or read-only API tokens: GitHub, GitLab, AWS, GCP, Azure, Jira, PagerDuty, etc. Platform pulls data via their APIs. No agent required.

Pros

  • Low friction — one click per tool
  • Works perfectly for cloud-native SaaS
  • Standard auth patterns sellers recognise

Cons

  • Doesn't reach on-prem or private network components
  • Dependent on what each API exposes
C

Hybrid — cloud connectors + optional agent

Cloud connectors handle the majority (GitHub, cloud infra, SaaS tooling). An on-prem agent is available for sellers with private network components, self-hosted tools, or on-prem databases.

Pros

  • Covers pure SaaS and hybrid/on-prem targets
  • Start with connectors, add agent only when needed
  • Matches the real diversity of target companies

Cons

  • More to build — two access paths to maintain
  • Connector library takes time to grow
